# CK 仓库作业系统（cangku）

这是一个面向海外仓一线作业的“扫码工时记录系统”。
当前系统已经可以在手机端通过扫码记录员工在不同业务、不同任务中的上岗/下岗事件，并支持全局在岗查看、管理员强制下岗、Session 管理等功能。

---

## 1) 你现在已经拥有了什么

### 前端（GitHub Pages）
- 入口页面：`index.html`
- 交互逻辑：`app.js`、`ui.js`
- 样式：`style.css`

前端作用：
- 让仓库主管/组长在手机端快速扫码。
- 记录 `start / join / leave / end / wave` 等关键事件。
- 在本地缓存必要状态（如当前 session/wave/名单）。

### 后端（Cloudflare Worker）
- 代码：`worker/index.js`
- 配置：`worker/wrangler.jsonc`

后端作用：
- 接收前端提交的事件。
- 写入 D1 数据库。
- 维护锁（Durable Object），避免同一工牌跨设备重复在岗。
- 提供管理员接口（查看会话、强制下岗、强制结束 session）。

---

## 2) 你最关心目标：工时 + WMS 人效分析

你的目标可以拆成三层：

1. **事实层（已在做）**：谁在什么时间，在哪个任务，做了多久。
2. **业务层（待对接）**：WMS 导出的每单/每波次作业结果（件数、行数、体积、完成时间等）。
3. **分析层（下一步重点）**：按人、按组、按任务、按天产出人效指标。

一句话：
**先把“人”和“时间”记录准，再把“产出”对上，就能形成稳定的人效报表。**

---

## 3) 建议的人效核心指标（第一版）

建议先做最小可用版本（MVP），避免一开始过复杂：

- 人员工时（小时）
  - 来源：同一人同一任务的 `join -> leave` 配对时长。
- 单位工时产出
  - 例：`拣货件数 / 拣货工时`、`订单数 / 打包工时`。
- 人均产出
  - 例：某任务在某天的人均件数、人均单量。
- 异常比率
  - 例：未配对事件（有 join 无 leave）、超长工时记录、跨设备冲突次数。

---

## 4) 数据治理规则（必须先定）

为了后续报表可信，强烈建议坚持以下规则：

1. **同一工牌在任一时刻只能有 1 条 active 在岗**（系统已有锁机制）。
2. **leave 必须对应最近一次 join**（不允许无 join 直接 leave）。
3. **跨天 session 必须有明确归属规则**（按 join 日期还是按 leave 日期）。
4. **任务编码固定化**（例如 PICK、PACK、RELABEL），不要随意改名字。
5. **WMS 导出字段固定模板**（后续自动化对接会依赖列名）。

## 5) 开发路线图（非技术人员可理解版）

详细版本请看：`docs/ROADMAP_CN.md`

### 阶段 A：把“工时记录”做扎实（1~2 周）
- 清洗事件配对逻辑。
- 输出每日“每人每任务工时”明细表。
- 增加异常日志（缺失 leave / 重复 join）。

### 阶段 B：接入 WMS 导出（1~2 周）
- 确认你现有 WMS 导出样本。
- 设计“字段映射表”。
- 生成“工时 + 产出”合并明细。

### 阶段 C：人效看板（2 周）
- 先做日报 CSV，再做图表。
- 维度：人 / 任务 / 组 / 时间。
- 增加目标值与预警（低于标准线标红）。
